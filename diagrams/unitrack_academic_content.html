Part One

Functional Code Snippets

This section presents the two core code implementations that handle student task submission across the full application stack: the React.js frontend component and the corresponding Node.js/Express backend controller.

Snippet A — React Component: Student Task Upload

The TaskUpload component provides the student-facing interface for submitting task deliverables. It uses React Hooks for local state management and the Axios library to send a multipart HTTP request to the backend API. Client-side validation enforces allowed file types and the 20 MB size ceiling before any network call is made.

JSX — React.js src/components/TaskUpload.jsx 
import React, { useState } from 'react';
import axios from 'axios';

/**
 * TaskUpload Component
 * Allows a student to upload a file for a specific task.
 *
 * Props:
 *   taskId   {number}   — ID of the task being submitted.
 *   onSuccess {Function} — Callback fired after successful upload.
 */
const TaskUpload = ({ taskId, onSuccess }) => {

  // ── Local State ──────────────────────────────────────────
  const [selectedFile, setSelectedFile] = useState(null);
  const [uploading,    setUploading]    = useState(false);
  const [progress,     setProgress]     = useState(0);
  const [message,      setMessage]      = useState('');
  const [isError,      setIsError]      = useState(false);

  // Permitted file extensions and corresponding MIME types
  const ALLOWED_TYPES = [
    'application/pdf',
    'application/msword',
    'application/vnd.openxmlformats-officedocument.wordprocessingml.document',
    'application/zip',
  ];
  const MAX_SIZE_BYTES = 20 * 1024 * 1024; // 20 MB

  // ── File Selection Handler ────────────────────────────────
  const handleFileChange = (e) => {
    const file = e.target.files[0];
    setMessage('');
    setIsError(false);

    if (!file) return;

    if (!ALLOWED_TYPES.includes(file.type)) {
      setMessage('Invalid file type. Please upload PDF, DOCX, or ZIP.');
      setIsError(true);
      return;
    }
    if (file.size > MAX_SIZE_BYTES) {
      setMessage('File is too large. Maximum allowed size is 20 MB.');
      setIsError(true);
      return;
    }
    setSelectedFile(file);
    setMessage(`File ready: ${file.name}`);
  };

  // ── Upload Handler ────────────────────────────────────────
  const handleUpload = async () => {
    if (!selectedFile) {
      setMessage('Please select a file before uploading.');
      setIsError(true);
      return;
    }

    // Build multipart form payload
    const formData = new FormData();
    formData.append('taskFile', selectedFile);
    formData.append('taskId',   taskId);

    // Retrieve stored JWT token for Authorization header
    const token = localStorage.getItem('auth_token');

    setUploading(true);
    setProgress(0);

    try {
      const response = await axios.post(
        `/api/tasks/${taskId}/upload`,
        formData,
        {
          headers: {
            'Authorization': `Bearer ${token}`,
            'Content-Type':  'multipart/form-data',
          },
          // Track real-time upload progress
          onUploadProgress: (event) => {
            const percent = Math.round((event.loaded * 100) / event.total);
            setProgress(percent);
          },
        }
      );

      setMessage(response.data.message || 'Task submitted successfully!');
      setIsError(false);
      setSelectedFile(null);
      if (onSuccess) onSuccess(response.data);

    } catch (error) {
      const errMsg = error.response?.data?.message
        || 'Upload failed. Please try again.';
      setMessage(errMsg);
      setIsError(true);
    } finally {
      setUploading(false);
    }
  };

  // ── Render ────────────────────────────────────────────────
  return (
    <div className="task-upload-container">

      <h3 className="upload-title">Submit Task File</h3>

      <input
        type="file"
        accept=".pdf,.doc,.docx,.zip"
        onChange={handleFileChange}
        disabled={uploading}
        className="file-input"
      />

      {/* Upload progress bar */}
      {uploading && (
        <div className="progress-bar-wrap">
          <div
            className="progress-bar-fill"
            style={{ width: `${progress}%` }}
          />
          <span>{progress}%</span>
        </div>
      )}

      {/* Status message */}
      {message && (
        <p className={isError ? 'msg-error' : 'msg-success'}>
          {message}
        </p>
      )}

      <button
        onClick={handleUpload}
        disabled={uploading || !selectedFile}
        className="btn-upload"
      >
        {uploading ? 'Uploading...' : 'Upload Task'}
      </button>

    </div>
  );
};

export default TaskUpload;
Key Design Decisions: Client-side MIME type checking is performed before the API request to improve user experience and reduce unnecessary server load. However, the backend independently validates the file type using server-side checks to prevent MIME spoofing. The JWT token is attached via the Authorization: Bearer <token> header, which the backend middleware verifies on every protected route.
Snippet B — Node.js / Express: File Upload Controller

The backend controller receives the multipart request, validates the uploaded file, saves it to the server's file system, persists the file path and task metadata in the MySQL database using Sequelize ORM, and returns a structured JSON response. A supervisor notification is dispatched asynchronously after successful save.

JavaScript — Node.js / Express controllers/taskController.js 
const multer  = require('multer');
const path    = require('path');
const { Task, Notification, Project } = require('../models');

/* ═══════════════════════════════════════════════════════
   MULTER — FILE STORAGE CONFIGURATION
   ═══════════════════════════════════════════════════════ */

const storage = multer.diskStorage({

  // Directory where uploaded files will be stored
  destination: (req, file, cb) => {
    cb(null, './uploads/tasks/');
  },

  // Generate a unique filename to prevent overwrite collisions
  filename: (req, file, cb) => {
    const timestamp = Date.now();
    const ext       = path.extname(file.originalname).toLowerCase();
    const safeName  = file.originalname
      .replace(ext, '')
      .replace(/[^a-zA-Z0-9_\-]/g, '_');
    cb(null, `task_${timestamp}_${safeName}${ext}`);
  },
});

// Server-side MIME type whitelist (prevents MIME spoofing)
const fileFilter = (req, file, cb) => {
  const allowed = [
    'application/pdf',
    'application/msword',
    'application/vnd.openxmlformats-officedocument.wordprocessingml.document',
    'application/zip',
  ];
  if (allowed.includes(file.mimetype)) {
    cb(null, true);   // Accept file
  } else {
    cb(new Error('INVALID_TYPE'), false); // Reject file
  }
};

// Export the configured upload middleware for route use
const upload = multer({
  storage,
  fileFilter,
  limits: { fileSize: 20 * 1024 * 1024 }, // 20 MB cap
});


/* ═══════════════════════════════════════════════════════
   CONTROLLER — uploadTaskFile
   Route:  POST /api/tasks/:taskId/upload
   Access: Authenticated Students only (JWT middleware)
   ═══════════════════════════════════════════════════════ */

const uploadTaskFile = async (req, res) => {
  try {

    // 1. Verify a file was received
    if (!req.file) {
      return res.status(400).json({
        success: false,
        message: 'No file provided. Please attach a file and try again.',
      });
    }

    const { taskId }   = req.params;
    const  studentId   = req.user.id;  // Injected by jwtAuthMiddleware

    // 2. Verify the task belongs to this student
    const task = await Task.findOne({
      where: { id: taskId, student_id: studentId },
    });

    if (!task) {
      return res.status(404).json({
        success: false,
        message: 'Task not found or you are not authorized to submit this task.',
      });
    }

    // 3. Reject if the task is already graded (no resubmission)
    if (task.status === 'graded') {
      return res.status(400).json({
        success: false,
        message: 'This task has already been graded and cannot be resubmitted.',
      });
    }

    // 4. Save file metadata to the MySQL database
    await task.update({
      file_path:    req.file.path,           // Relative path on server
      file_name:    req.file.originalname,   // Original filename for display
      file_size:    req.file.size,           // Size in bytes
      status:       'submitted',           // Transition task state machine
      submitted_at: new Date(),            // Record submission timestamp
    });

    // 5. Create a notification for the project supervisor (async)
    const project = await Project.findByPk(task.project_id);
    if (project?.supervisor_id) {
      Notification.create({
        user_id:         project.supervisor_id,
        message:         `Student ${req.user.name} submitted task: "${task.title}".
                         Please review the submission.`,
        type:            'task_submitted',
        notifiable_type: 'Task',
        notifiable_id:   task.id,
        is_read:         false,
      }).catch(console.error); // Non-blocking — do not fail the request
    }

    // 6. Return structured success response
    return res.status(200).json({
      success:  true,
      message:  'Task file uploaded and saved successfully.',
      data: {
        task_id:      task.id,
        file_name:    req.file.originalname,
        file_size_kb: Math.round(req.file.size / 1024),
        status:       task.status,
        submitted_at: task.submitted_at,
      },
    });

  } catch (error) {
    console.error('[uploadTaskFile ERROR]', error.message);
    return res.status(500).json({
      success: false,
      message: 'An internal server error occurred. Please try again later.',
    });
  }
};

module.exports = { upload, uploadTaskFile };
Route Registration Example: In the Express router file, the controller is wired as: router.post('/tasks/:taskId/upload', jwtAuth, upload.single('taskFile'), uploadTaskFile); The jwtAuth middleware verifies the token and attaches req.user before the controller runs.
Part Two

Database Table Descriptions

The following tables form the core of the UniTrack AI MySQL relational database. The schema is normalized to Third Normal Form (3NF) to eliminate redundancy and enforce referential integrity.

Table 1 — users

Stores all platform accounts. Role-based access is controlled via a foreign key link to the roles table, enabling the system to serve four distinct user types from a single entity.

TABLE: users
FieldTypeDescription
id PK BIGINT UNSIGNED
AUTO_INCREMENT Unique numeric identifier for each user. Referenced as a foreign key throughout all other tables that associate data with a specific user.
name VARCHAR(255)
NOT NULL The user's full legal name, displayed across dashboards, task views, and report outputs.
email UNIQUE VARCHAR(255)
NOT NULL Primary login identifier and contact address. Enforced unique at the database level to prevent duplicate account registration.
password VARCHAR(255)
NOT NULL Bcrypt-hashed user password. Raw passwords are never stored; hashing is performed at the application layer prior to any database write.
role_id FK BIGINT UNSIGNED
NOT NULL Foreign key referencing roles.id. Determines which dashboard, routes, and actions are accessible to the user (student, supervisor, assistant, admin).
is_active TINYINT(1)
DEFAULT 1 Account activation flag. When set to 0 by an administrator, the user cannot log in. Supports suspension without permanent deletion of audit history.
last_login_at DATETIME
NULLABLE Timestamp of the user's most recent successful login. Used by the admin panel to identify inactive accounts.
created_at TIMESTAMP Record creation timestamp, set automatically by the database engine at insert time.
updated_at TIMESTAMP Automatically updated to the current time whenever any field in the user record is modified.
Table 2 — projects

Represents graduation projects registered on the platform. Each project is supervised by one faculty member and may optionally include a teaching assistant. The status field governs the project's lifecycle.

TABLE: projects
FieldTypeDescription
id PK BIGINT UNSIGNED
AUTO_INCREMENT Unique project identifier. Referenced by the tasks table to associate deliverables with their parent project.
title VARCHAR(500)
NOT NULL The official title of the graduation project as registered by the supervisor. Displayed in all listing and reporting interfaces.
description LONGTEXT
NULLABLE Full project scope, objectives, and methodology. Used by the AI Copilot to generate context-aware resource suggestions.
supervisor_id FK BIGINT UNSIGNED
NOT NULL Foreign key referencing users.id for the supervising professor. This user receives all task submission notifications and controls grading.
status ENUM
DEFAULT 'pending' Project lifecycle state. Valid values: pending, active, completed, archived. Drives conditional UI rendering and automated cron-based transitions.
start_date DATE
NULLABLE Official project start date used as the baseline for progress calculations and timeline charts.
end_date DATE
NULLABLE Project submission deadline. When this date passes, the system triggers a status transition to completed via a scheduled cron job.
created_at TIMESTAMP Project registration timestamp, set automatically at insert time.
updated_at TIMESTAMP Automatically updated on any field modification, providing a complete audit trail for project changes.
Table 3 — tasks

The operational core of the system. Each record represents a deliverable that a student must submit within a project. The table maintains the full history of submission, review, grading, and feedback cycles.

TABLE: tasks
FieldTypeDescription
id PK BIGINT UNSIGNED
AUTO_INCREMENT Unique task record identifier.
project_id FK BIGINT UNSIGNED
NOT NULL Foreign key to projects.id. Groups tasks under their parent project for aggregated progress reporting.
student_id FK BIGINT UNSIGNED
NOT NULL Foreign key to users.id identifying the student responsible for this task submission.
title VARCHAR(255)
NOT NULL Descriptive name of the task (e.g., "Phase 2 — System Design Document").
file_path VARCHAR(1000)
NULLABLE Server-side path to the uploaded deliverable file. NULL indicates the task has not yet been submitted. Updated by the upload controller on each submission.
file_name VARCHAR(500)
NULLABLE The original filename as provided by the student's device, preserved for display purposes in the supervisor review interface.
grade DECIMAL(5,2)
NULLABLE Numerical grade awarded by the supervisor on a 0.00–100.00 scale. A NULL value indicates the task is pending evaluation.
feedback TEXT
NULLABLE Written qualitative feedback from the supervisor. Displayed to the student after grading and retained permanently in the academic record.
status ENUM
DEFAULT 'pending' Task lifecycle state. Valid values: pending, submitted, under_review, graded, resubmit_required. Drives dashboard status badges and notification triggers.
due_date DATETIME
NULLABLE Submission deadline set by the supervisor. The cron job compares this against current time to dispatch deadline reminder notifications.
submitted_at DATETIME
NULLABLE Exact timestamp of the student's file upload. Set once on first successful submission; immutable thereafter to preserve the audit record.
graded_at DATETIME
NULLABLE Timestamp when the supervisor finalized and saved the grade, completing the evaluation audit trail.
created_at / updated_at TIMESTAMP Standard audit timestamps managed automatically by the database engine.
Table 4 — notifications

Records all system-generated and user-triggered notifications. Each notification is scoped to a single recipient. The is_read flag powers the unread badge counter displayed across all dashboard headers.

TABLE: notifications
FieldTypeDescription
id PK BIGINT UNSIGNED
AUTO_INCREMENT Unique notification record identifier.
user_id FK BIGINT UNSIGNED
NOT NULL Foreign key to users.id identifying the notification recipient. Indexed for high-performance unread-count queries executed on every dashboard load.
message TEXT
NOT NULL The human-readable notification body rendered in the notification panel (e.g., "Your task 'System Design' has been graded: 91/100").
type ENUM
NOT NULL Categorizes the notification for icon and filter-by-type rendering. Values: task_submitted, grade_received, feedback_added, deadline_reminder, system.
notifiable_type VARCHAR(100)
NULLABLE Polymorphic entity reference (e.g., 'Task', 'Project'). Enables deep-linking from the notification to the relevant detail page.
notifiable_id BIGINT UNSIGNED
NULLABLE The ID of the referenced entity (paired with notifiable_type). Together they form a polymorphic relationship that links the notification to its source object.
is_read TINYINT(1)
DEFAULT 0 Read-state flag. 0 = unread (included in badge counter). 1 = read (dismissed). Updated in bulk when the user opens the notification panel.
read_at DATETIME
NULLABLE Exact timestamp of when the user acknowledged the notification, recorded for engagement analytics.
created_at TIMESTAMP Notification dispatch timestamp. Used to sort notifications in reverse chronological order (most recent first).
