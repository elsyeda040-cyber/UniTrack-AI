Chapter 3: Logical Design — Database Schema
Detailed MySQL relational schema, field definitions, constraints, and entity relationships
3.1 Database Overview

The UniTrack AI platform adopts a normalized relational database design using MySQL v8.x, organized into five core entities that collectively govern all platform operations — from user authentication to grade management.

The schema follows Third Normal Form (3NF) to eliminate data redundancy and enforce referential integrity through foreign key constraints. All primary keys use unsigned BIGINT AUTO_INCREMENT identifiers unless otherwise specified. Timestamps are recorded as DATETIME fields with server-side defaults to ensure audit trail compliance.

ℹ️
Naming Convention: All table names are lowercase plural nouns (e.g., users, tasks). Foreign keys follow the pattern {referenced_table_singular}_id. Boolean fields use TINYINT(1) with default value of 0.
3.2 Table: users

The central entity of the system. Every individual interacting with the platform — whether a student, supervisor, assistant, or administrator — is represented as a record in this table. Role-based access control (RBAC) is enforced via a foreign key reference to the roles table.

4.2 System Architecture — Client-Server Model

UniTrack AI follows a decoupled three-tier architecture separating the Presentation Layer (React.js), the Application Logic Layer (Node.js/Express or Laravel), and the Data Layer (MySQL).

Communication between the frontend and backend is entirely mediated through a RESTful API over HTTPS. The React client makes asynchronous HTTP requests via the Axios library and processes JSON-encoded responses to update the UI state reactively. Authentication is stateless, implemented via JSON Web Tokens (JWT) injected into every authenticated request via Axios interceptors.

🖥️ Presentation Tier — React.js

Single Page Application (SPA) built with React 18, Tailwind CSS, and Vite. Routes are protected by role-aware guards. State management uses React Context API + localStorage for token persistence.

⚙️ Application Tier — Node.js / Laravel

REST API server exposing JSON endpoints at /api/v1/*. Handles business logic, authentication middleware, file upload processing, and notification triggers. Deployed on Railway.

🗄️ Data Tier — MySQL

Relational database with 13 tables. Accessed exclusively through the ORM layer (Sequelize for Node.js, Eloquent for Laravel), preventing raw SQL injection vectors.

🤖 AI Tier — Google Gemini API

External intelligence layer called from the backend to power the AI Copilot, meeting summarization, team health scoring, and learning resource extraction features.

🔒
Security Architecture: All API routes except /api/auth/login and /api/auth/register require a valid JWT bearer token. Role-based middleware further restricts resource access — for example, only supervisor and admin roles can access grade submission endpoints.
4.3 Code Snippet A — React Task Upload Component

The following functional component implements the student task file upload workflow using the Axios HTTP client. It manages multi-state feedback (idle, uploading, success, error) and enforces client-side file type validation before dispatching the multipart form submission to the backend API.

JSX — React.js src/components/TaskUpload.jsx 
import React, { useState, useRef } from 'react';
import axios from 'axios';

/**
 * TaskUpload — Handles file submission for a given task.
 * @param {string} taskId — The unique ID of the task to upload to.
 * @param {Function} onSuccess — Callback invoked on successful upload.
 */
const TaskUpload = ({ taskId, onSuccess }) => {
  const [file, setFile] = useState(null);
  const [status, setStatus] = useState('idle'); // idle | uploading | success | error
  const [progress, setProgress] = useState(0);
  const [errorMsg, setErrorMsg] = useState('');
  const fileInputRef = useRef(null);

  // Permitted MIME types for task submissions
  const ALLOWED_TYPES = [
    'application/pdf',
    'application/msword',
    'application/vnd.openxmlformats-officedocument.wordprocessingml.document',
    'application/zip',
  ];

  /** Validate and stage the selected file */
  const handleFileSelect = (e) => {
    const selected = e.target.files[0];
    if (!selected) return;

    // Client-side MIME type validation
    if (!ALLOWED_TYPES.includes(selected.type)) {
      setErrorMsg('Invalid file type. Allowed: PDF, DOCX, ZIP.');
      return;
    }
    // Max file size: 20 MB
    if (selected.size > 20 * 1024 * 1024) {
      setErrorMsg('File exceeds maximum size of 20 MB.');
      return;
    }
    setFile(selected);
    setErrorMsg('');
  };

  /** Submit file to the REST API using multipart/form-data */
  const handleSubmit = async () => {
    if (!file) { setErrorMsg('Please select a file before submitting.'); return; }
    setStatus('uploading');
    const formData = new FormData();
    formData.append('file', file);
    formData.append('task_id', taskId);

    try {
      const token = localStorage.getItem('auth_token');
      await axios.post(
        `/api/v1/tasks/${taskId}/submit`,
        formData,
        {
          headers: {
            'Authorization': `Bearer ${token}`,
            'Content-Type': 'multipart/form-data',
          },
          onUploadProgress: (evt) => {
            const pct = Math.round((evt.loaded * 100) / evt.total);
            setProgress(pct);
          },
        }
      );
      setStatus('success');
      onSuccess?.(); // Notify parent to refresh task list
    } catch (err) {
      setStatus('error');
      setErrorMsg(err.response?.data?.message || 'Upload failed. Please try again.');
    }
  };

  return (
    <div className="upload-container">
      <input ref={fileInputRef} type="file"
        accept=".pdf,.doc,.docx,.zip"
        onChange={handleFileSelect} />
      {status === 'uploading' && <div>Uploading... {progress}%</div>}
      {errorMsg && <p className="text-red-600">{errorMsg}</p>}
      {status === 'success' && <p>✅ Task submitted successfully!</p>}
      <button onClick={handleSubmit}>Submit Task</button>
    </div>
  );
};

export default TaskUpload;
📌
Security Note: Client-side MIME type validation is a UX enhancement only. Server-side validation (magic byte inspection) is mandatory and implemented independently in the backend controller to prevent MIME spoofing attacks.

const multer  = require('multer');

/* ─── Multer Storage Configuration ─── */
const storage = multer.diskStorage({
  destination: (req, file, cb) => {
    cb(null, 'uploads/tasks/'); // Absolute path in production
  },
  filename: (req, file, cb) => {
    // Timestamp + sanitized original name to prevent collisions

    cb(null, `${timestamp}_${sanitized}`);

const fileFilter = (req, file, cb) => {
  const allowedMimes = [
    'application/pdf', 'application/msword',
    'application/vnd.openxmlformats-officedocument.wordprocessingml.document',
    'application/zip',
  ];
  allowedMimes.includes(file.mimetype)
    ? cb(null, true)      // Accept
    : cb(new Error('INVALID_FILE_TYPE'), false); // Reject
};

const upload = multer({ storage, fileFilter, limits: { fileSize: 20 * 1024 * 1024 } });

Implementation B — Laravel / PHP

PHP — Laravel 11 app/Http/Controllers/TaskController.php 
namespace App\Http\Controllers;

use App\Models\Task;
use Illuminate\Http\Request;
use Illuminate\Support\Facades\Storage;
use App\Events\TaskSubmitted; // Triggers notification broadcast

class TaskController extends Controller
{
    /**
     * Handle an authenticated student's task file submission.
     * Route: POST /api/v1/tasks/{taskId}/submit
     * Middleware: auth:sanctum, role:student
     */
    public function submit(Request $request, int $taskId): \Illuminate\Http\JsonResponse
    {
        // 1. Validate uploaded file at the framework level
        $request->validate([
            'file' => [
                'required', 'file',
                'mimes:pdf,doc,docx,zip',
                'max:20480', // 20 MB in kilobytes
            ],
        ]);

        // 2. Authorize: Ensure the task belongs to the authenticated student
        $task = Task::where('id', $taskId)
                    ->where('student_id', $request->user()->id)
                    ->firstOrFail(); // Throws 404 if not found

        // 3. Delete previous submission file if it exists
        if ($task->file_path) {
            Storage::disk('tasks')->delete($task->file_path);
        }

        // 4. Store the new file with a sanitized hashed name
        $file     = $request->file('file');
        $filePath = $file->store(
            "tasks/{$task->project_id}",
            'tasks' // Disk defined in config/filesystems.php
        );

        // 5. Persist metadata and transition task state machine
        $task->update([
            'file_path'    => $filePath,
            'file_name'    => $file->getClientOriginalName(),
            'file_size'    => $file->getSize(),
            'status'       => 'submitted',
            'submitted_at' => now(),
        ]);

        // 6. Fire event to trigger supervisor notification
        event(new TaskSubmitted($task));

        return response()->json([
            'message' => 'Task submitted successfully.',
            'task'    => $task->only(['id', 'status', 'submitted_at', 'file_name']),
        ], 200);
    }
}

4.7 Real-Time Notification Workflow

The notification subsystem is event-driven. Each domain action (task submission, grading, deadline proximity) triggers a corresponding notification record insertion. The following workflow diagram documents the technical execution path:

1
Action Trigger

A domain event occurs — e.g., a student submits a task via POST /api/v1/tasks/{id}/submit. The controller updates the task record and status transitions to submitted.

2
Event Dispatch

The controller fires a domain event (TaskSubmitted in Laravel / EventEmitter in Node.js). This decouples notification logic from the request-response cycle, preventing API latency spikes.

3
Listener Execution

The registered event listener resolves the notification recipients (supervisor + assistant for the project). It constructs the notification message template with interpolated context data.

4
Database Persistence

The listener performs a bulk INSERT into the notifications table — one record per recipient — with is_read = 0 and the appropriate notifiable_type / notifiable_id polymorphic references.

5
Client-Side Polling / WebSocket Push

The supervisor's dashboard polls GET /api/v1/notifications/unread-count every 30 seconds. When the count increases, the notification bell badge updates reactively. Optional WebSocket integration (Laravel Echo / Socket.io) enables instant push delivery.

6
Read Acknowledgment

When the user opens the notification panel, a PATCH /api/v1/notifications/read-all request sets is_read = 1 and read_at = NOW() for all unread records belonging to the authenticated user.

4.8 Automated Progress Tracking

The platform calculates project progress automatically using a multi-factor formula that considers task completion rate, submission timeliness, and average grade performance. This score is surfaced in real-time on both student and supervisor dashboards.

📊
Progress Calculation Formula:
Project Progress (%) = (Completed Tasks / Total Tasks) × 100

Task Completion Score:
A task is classified as "completed" when its status = graded AND grade ≥ passing_threshold (default: 50.00).

Gamification Score:
Score = (Completed × 50) + (Average Grade × 5) + (Achievement Badges × 100)

5.1 Testing Strategy Overview

The quality assurance strategy for UniTrack AI encompasses three levels of testing: Unit Testing (individual API endpoints), Integration Testing (cross-component workflows), and User Acceptance Testing (UAT) performed against the deployed staging environment.

All API endpoint tests were executed using Postman Collections. Frontend component behavior was validated manually and through browser DevTools inspection. The following test plan documents the black-box test cases covering all primary system features.

🔬 Unit Testing

Each API controller function was tested in isolation with mocked database responses using Jest (Node.js) or PHPUnit (Laravel). Coverage target: 80%+ for critical business logic paths.

🔗 Integration Testing

End-to-end workflow tests validated the complete task submission path: React component → Axios request → API controller → Database → Notification trigger → Dashboard update.

🖥️ UI / Client Testing

Manual testing across Chrome, Firefox, and Edge browsers. Responsive layout was validated at breakpoints: 375px (mobile), 768px (tablet), 1280px (desktop).

🛡️ Security Testing

Authentication bypass, SQL injection (via ORM parameterization), MIME type spoofing, and unauthorized role escalation were all tested and confirmed protected.
